# 任务系统 — qa1 测试用例输入

> 来源: pm1 的任务系统规范文档 (docs/task-system-spec.md)
> 目标: 验证当前 Solo 任务系统实现是否符合规范
> 工具: Playwright + API 测试

---

## P0 测试用例 (必须通过)

### TC-01: 全局任务列表加载

- **前置**: 用户已登录
- **操作**: 导航到 `/tasks`
- **验证**:
  1. 页面标题显示 "任务列表"
  2. 如果有任务，显示卡片网格
  3. 如果没有任务，显示空态 "暂无任务" + "创建任务" 按钮
  4. 加载中有骨架屏
  5. 加载失败有错误提示和 "重试" 按钮

### TC-02: 创建任务 (全局页面)

- **前置**: 用户已登录，存在至少一个频道
- **操作**: 导航到 `/tasks/new`
- **验证**:
  1. 页面显示 "创建任务" 标题
  2. 频道选择下拉列表可选
  3. 标题必填，空标题提交被拒绝
  4. 填写完整表单后提交 → 创建成功
  5. 创建成功后跳转到 `/tasks`

### TC-03: 创建任务 (频道内 quick-create)

- **前置**: 用户已登录，打开频道
- **操作**: 在频道 Messages/Tasks Tab 中点击 "创建任务"
- **验证**:
  1. 弹出 Modal，标题为 "快速创建任务"
  2. 频道字段已预选（不可编辑）
  3. 填写表单后创建成功
  4. Modal 关闭，任务出现在频道 Tasks Tab 中

### TC-04: 任务详情加载

- **前置**: 存在任务
- **操作**: 点击任务卡片 → 跳转到 `/tasks/{id}`
- **验证**:
  1. 页面显示完整任务信息: 标题、描述、状态 Badge、优先级 Badge、指派人、频道、创建时间
  2. 状态转换按钮根据当前状态正确展示
  3. 评论区域展示
  4. 任务不存在时显示 "任务不存在" + 返回列表 + 重试

### TC-05: 任务状态流转

- **前置**: 存在一个 todo 状态的任务
- **操作**: 在任务详情页点击状态转换按钮
- **验证**:
  1. todo → "开始处理" → 状态变为 in_progress (黄色)
  2. in_progress → "提交审查" → 状态变为 in_review (紫色)
  3. in_review → "标记完成" → 状态变为 done (绿色)
  4. done → "重新打开" → 状态变为 in_progress
  5. todo → "取消任务" → 状态变为 cancelled (红色)
  6. cancelled → "重新打开" → 状态变为 todo
  7. 使用 API 直接测试非法转换 (如 done → todo) 返回 400

### TC-06: 任务筛选

- **前置**: 存在多个不同状态的任务
- **操作**: 在 `/tasks` 页面使用筛选下拉框
- **验证**:
  1. 按状态筛选: 选择 "进行中" → 只显示 in_progress 任务
  2. 按指派人筛选: 选择指派人 → 只显示该指派人的任务
  3. 筛选后可重置为 "全部状态"

### TC-07: 任务指派

- **前置**: 存在频道成员 (用户和 Agent)
- **操作**: 创建任务时选择指派人
- **验证**:
  1. 指派人下拉列表包含频道中的用户和 Agent
  2. 指派后任务详情显示指派人名称
  3. 创建后可以重新指派 (任务详情页 "重新指派" 按钮)

### TC-08: 任务删除

- **前置**: 当前用户创建了至少一个任务
- **操作**: 通过 API DELETE /api/v1/tasks/{id}
- **验证**:
  1. 删除成功返回 204
  2. 删除后 GET /api/v1/tasks/{id} 返回 404
  3. 删除后任务列表不再显示该任务

### TC-09: 侧栏任务入口

- **前置**: 用户已登录
- **操作**: 查看侧栏
- **验证**:
  1. 侧栏底部有 "任务管理" 链接 (ClipboardList 图标)
  2. 点击后导航到 `/tasks`
  3. Agent 管理链接也存在 (Bot 图标)

- **操作**: 创建任务并指派给 Agent

## 测试步骤说明

1. **API 测试**: 使用 Playwright 的 APIRequestContext 直接调用后端 API
2. **UI 测试**: 使用 Playwright page 对象进行浏览器交互测试
3. **WS 测试**: 使用 Playwright 的 WebSocket route 拦截或监听 WS 事件

优先验证 P0 用例，P0-新功能 用例建议先确认后端 API 是否实现再做回归。
