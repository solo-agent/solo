## 1. 产品定位与愿景

### 产品定位

Solo 是一个**频道式实时多 Agent 协作平台**。人类用户和 AI Agent 可以在频道（Channel）中实时协作，Agent 可以"看到"频道上下文并主动参与对话，人类可以通过简单的频道机制编排多个 Agent 协同工作。

### 一句话描述

> 像 Slack 一样组织人+AI 的协作空间，但每个频道里不仅有同事，还有 AI Agent。

### 核心价值

1. **打破 AI 的"对话框孤岛"** — 每个 Agent 不再是单独的聊天窗口，而是团队协作空间中的平等成员
2. **Agent 之间的可见协作** — 多个 Agent 在同一频道中工作，人类可以观察、干预、引导
3. **上下文共享** — 所有频道成员（人和 AI）共享频道消息历史，Agent 能理解对话上下文
4. **低门槛上手** — 频道结构和 Slack 一致，无需学习新概念

### 产品愿景

让任何团队都能在自己的协作空间中编排 AI Agent，像管理团队成员一样管理 AI 工作者，实现人机协作效率最大化。

## 2. 目标用户画像

### 主要用户群

| 画像 | 描述 | 核心场景 |
|------|------|----------|
| **独立开发者** | 个人开发者，使用多个 AI 工具处理不同任务 | 在频道中同时使用代码审查 Agent 和文档 Agent |
| **小团队** | 2-10 人的技术/产品团队 | 团队频道中加入 QA Agent 自动测试 PR |
| **AI 爱好者/研究者** | 实验多 Agent 协作模式 | 搭建多个不同角色的 Agent 观察它们协作 |
| **知识工作者** | 需要 AI 辅助研究、写作、数据分析 | 在研究频道中加入多个专业 Agent |

### 非目标用户（MVP 阶段）

- 大型企业组织（100+ 人）— 暂不需要多租户、SSO、组织层级
- 非技术用户 — 产品初期以技术用户为主，UI/UX 复杂度可接受
- 需要严格合规/审批流程的行业 — 审批流放后续版本

### 用户核心痛点

1. 现有 AI 工具是单人对话模式，无法在团队上下文中协作
2. 多个 AI 工具之间无法共享上下文和工作成果
3. 缺少一个统一的"AI 团队协作工作台"
4. 编排多个 Agent 协作的复杂度高，缺乏合适的界面

## 3. 竞品对标：slock.ai

### slock.ai 核心体验分析

Slock.ai 是"频道式多 Agent 协作"赛道的领先产品，其核心体验包括：

| 功能 | 描述 | Solo MVP 是否对标 |
|------|------|-------------------|
| 频道（Channels） | 基于主题的协作空间，人和 Agent 共存 | **是** — 核心模式 |
| 私信（DM） | 1对1 私密对话 | **是** — MVP 核心 |
| 线程（Threads） | 消息的回复线程 | **是** — 核心模式 |
| Agent 市场 | 预配置的 Agent 模板 | 否 — MVP 后 |
| Agent 自定义 | 配置 prompt、模型、工具 | **是** — 核心模式 |
| 实时流式输出 | Agent 生成内容实时可见 | **是** — 核心体验 |
| @提及 Agent | 在消息中 @agent 触发响应 | **是** — 核心模式 |
| 自动加入频道 | Agent 自动监听频道消息 | **是** — 核心模式 |
| 消息编辑/删除 | 编辑和删除已发消息 | **是** — 核心模式 |
| 文件上传 | 在消息中附加文件 | 否 — MVP 后 |
| 消息搜索 | 全文搜索消息历史 | 否 — MVP 后 |

### Solo 的差异化

1. **开源** — 用户可以自托管，数据完全掌控
2. **模型无关** — 支持 OpenAI、Anthropic、Ollama 等多种 LLM，用户自选
3. **技术栈复用 multica** — 站在成熟产品基础上构建，不是从零开始
4. **更简化的定价** — 自有 API Key 模式，无需按席位付费

## 6. MVP 范围定义

### MVP 的核心闭环

```
核心消息流:
用户注册 -> 创建频道 -> 创建 Agent -> 将 Agent 加入频道
-> 在频道中发消息 -> Agent 自动响应（流式输出） -> 所有成员实时看到

线程回复流:
点击消息 "回复" -> 线程面板在右侧展开 -> 在线程中输入回复
-> Agent 在线程上下文中响应 -> 所有成员实时看到线程消息

@提及 触发流:
在消息输入框中输入 @ -> 弹出 Agent/用户选择器 -> 选中目标 Agent
-> 发送消息 -> 被 @提及 的 Agent 被触发 -> Agent 响应出现在频道中

私信(DM)流:
侧栏点击 "新私信" -> 选择收信人（用户或 Agent）-> 创建 DM 频道
-> 在 DM 中发送消息 -> 对方实时接收 -> 支持线程回复
```

### MVP 包含的功能（P0）

| 模块 | 包含范围 | 不包含 |
|------|---------|--------|
| 认证 | 邮箱注册/密码登录/登出 | OAuth、SSO、多因素认证 |
| 频道 | 创建、查看频道列表、进入频道、消息列表、发送消息、私信(DM)频道类型 | 频道分类、频道归档 |
| 消息 | 纯文本消息、实时推送、消息历史（游标分页）、线程回复 | 富文本、文件上传、消息搜索 |
| Agent | 创建 Agent（配置名称/模型/system prompt）、启停 | 工具配置、技能系统、知识库 |
| Agent 协作 | Agent 自动监听并响应频道消息、流式输出、@提及触发 Agent | 多 Agent 编排、审批流 |
| WebSocket | 频道级别的实时消息广播、Agent 状态推送 | 跨实例广播、离线消息 |
| 用户 | 基本个人资料 | 头像上传、偏好设置 |
| 界面 | 左栏频道列表 + DM 列表 + 右栏当前频道消息视图 + 线程面板 | Agent 市场 |

### MVP 刻意不做的事情

- Agent 工具配置（Agent 默认无工具，只做对话）
- 消息编辑/删除
- 用户头像上传
- 多模型切换（MVP 固定使用一个模型）
- 用量统计和预算
- 文件上传和附件
- 消息搜索
- 审批流
- 定时任务
- 插件系统
- Agent 记忆/RAG
- 多语言
- 暗色模式
- 移动端适配

## 7. 用户核心使用流程

### 完整使用路径（Happy Path）

```
Step 1: 注册/登录
  └─ 访问 /auth/register → 填写邮箱+密码 → 自动登录 → 进入 Dashboard

Step 2: Dashboard 首页（空态）
  └─ 左侧显示空频道列表 + "创建第一个频道" 引导按钮
  └─ 右侧显示欢迎页 / 空白占位

Step 3: 创建频道
  └─ 点击"创建频道" → 弹出表单
  └─ 填写频道名称（必填）+ 描述（选填）
  └─ 点击创建 → 频道出现在左侧列表 → 自动进入该频道

Step 4: 新频道空态
  └─ 频道内显示"这是 #频道名 的开端"占位
  └─ 消息列表为空（如果有系统消息则显示）
  └─ 提示"添加 Agent 来开始协作"

Step 5: 创建 Agent
  └─ 点击顶部 "+" 或 Agent 管理按钮 → 进入创建 Agent 页面
  └─ 填写 Agent 名称（必填）+ 系统提示词（选填，有默认值）
  └─ 选择模型（MVP 阶段只显示一个默认模型）
  └─ 点击创建 → Agent 创建成功

Step 6: 将 Agent 加入频道
  └─ 在频道设置/成员管理中 → 添加成员
  └─ 选择 "Agent" 类型 → 选中刚创建的 Agent
  └─ Agent 显示在频道成员列表中

Step 7: 发送消息
  └─ 在频道输入框中输入消息 → 回车发送
  └─ 消息出现在消息列表中，实时推送给所有在线成员

Step 8: Agent 响应
  └─ 系统检测到新消息 → 触发频道中的 Agent
  └─ Agent 状态变为 "thinking..." / 打字指示器
  └─ Agent 开始生成回复 → 流式逐块输出到频道
  └─ Agent 回复完整展示 → 完成

Step 9: 持续对话
  └─ 人类继续发送消息
  └─ Agent 每次看到新消息 → 基于完整上下文回复
  └─ 所有参与者实时看到对话

Step 10: 线程回复（Thread Reply）
  └─ 任意消息上悬停 → 点击 "回复" 按钮（显示为对话气泡图标）
  └─ 右侧线程面板打开 → 显示该消息的线程消息列表
  └─ 在线程底部输入框中输入回复 → 回车发送
  └─ 线程中的 Agent 自动响应（基于线程上下文，而非频道上下文）
  └─ 主频道消息旁显示 "N 条回复" 指示器
  └─ 点击线程外区域可关闭线程面板

Step 11: @提及 Agent
  └─ 在频道输入框中输入 @ 字符
  └─ 弹出自动完成下拉列表（包含频道中的 Agent 和用户）
  └─ 继续输入 → 列表过滤匹配的成员
  └─ 点击或键盘选择目标 Agent（如 @CodeReviewer）
  └─ 发送消息 → 被 @提及 的 Agent 被单独触发
  └─ Agent 回复时知道它是被 @提及 触发的（可以据此调整响应行为）
  └─ 如果频道中有 Auto-Respond Agent，只有 @提及 的 Agent 响应（其他 Agent 不触发）

Step 12: 私信（Direct Message）
  └─ 左侧栏底部有 "私信（DM）" 区域
  └─ 点击 DM 区域右上角的 "+" 按钮 → 弹出 "新私信" 对话框
  └─ 搜索/选择收信人（可以选用户或 Agent）
  └─ 点击选定 → 创建 DM 频道 → 自动进入该 DM 视图
  └─ DM 视图与频道视图基本一致：消息列表 + 输入框
  └─ 发送消息 → 对方实时接收
  └─ 支持线程回复和 @提及（与频道行为一致）
  └─ 再次与同一人私信 → 进入已有 DM 频道，不会重复创建
```

### 异常流程

| 场景 | 处理方式 |
|------|----------|
| 注册时邮箱已存在 | 提示"该邮箱已注册，请登录" |
| 登录失败 | 提示"邮箱或密码错误" |
| 创建频道名称重复 | 提示"该频道名称已存在" |
| WebSocket 断线 | 客户端显示"重新连接中..."，自动重连，重连后拉取最新消息 |
| Agent 响应超时 | 显示"Agent 响应超时，请重试" |
| Agent LLM 调用失败 | 显示"Agent 遇到了错误：[错误信息]" |
| 消息发送失败 | 本地暂存，重试发送，重试失败提示用户 |
| 删除有消息的频道 | 二次确认："删除后所有消息将丢失" |

## 8. 功能详述

### 8.1 F-01: 认证系统

**概述**: 支持用户通过邮箱+密码注册和登录。认证状态通过 JWT 维护。

**交互流程**:
1. 未认证用户访问任何页面 → 跳转到 `/auth/login`
2. 用户选择"注册" → `/auth/register` 表单
3. 用户选择"登录" → `/auth/login` 表单
4. 登录成功后 → 页面跳转到 `/dashboard`，后端返回 access token (15min) + refresh token
5. Token 过期 → 自动 refresh，失败则强制跳转到登录页

**业务规则**:
| 规则 | 内容 |
|------|------|
| 邮箱格式 | 标准 email 格式校验 |
| 密码强度 | 最少 8 位 |
| 密码存储 | bcrypt 哈希 |
| 会话管理 | JWT access token (15min) + refresh token 机制 |
| 登录限制 | 无（MVP 暂不限制） |

### 8.5 F-07/F-08: Agent 自动响应与流式输出

**概述**: 当频道中有新消息时，Agent 自动检测并生成回复。Agent 的回复通过流式输出实时推送到频道，用户可以看到 Agent 逐字生成内容。

**交互流程**:
1. 用户发送消息到频道
2. 服务端判断消息是否需要触发 Agent（MVP: 所有普通消息都触发）
3. 服务端向 Agent Daemon 下发任务
4. Daemon 调用 LLM API，附带频道最近 N 条消息作为上下文
5. LLM 流式返回，Daemon 通过 WebSocket 逐块推送到频道
6. 前端实时显示 Agent 的生成内容（Streaming Markdown）
7. 完成后，完整消息持久化到数据库

**触发逻辑（MVP）**:
- **默认行为**: 频道中所有新消息（用户发送）都触发所有活跃 Agent
- **触发间隔**: 同一 Agent 在 2 秒内不会被重复触发（防抖动）
- **上下文**: 传递给 Agent 最近 20 条消息（含当前触发消息）
- **空回复**: 如果 Agent 决定不回复（如仅问候场景），系统不做特殊处理

**流式输出 UI**:
- Agent 开始生成 → 输入框上方显示 "AgentName 正在输入..."
- 生成内容逐字出现 → 消息卡片中实时更新
- 支持 Markdown 渲染（代码块、列表、粗体等）
- 生成完成后 → 稳定显示完整消息

**Agent 状态指示**:
| 状态 | 显示 | 说明 |
|------|------|------|
| 在线/空闲 | 绿色圆点 | Agent 在监听但未执行任务 |
| 思考中 | 黄色圆点 + "thinking..." | Agent 正在调用 LLM |
| 输出中 | 蓝色圆点 + "typing..." | Agent 正在流式输出 |
| 离线/错误 | 灰色圆点 + "unavailable" | Agent 无法连接或有错误 |
| 已禁用 | 红色圆点 | Agent 被用户手动暂停 |

**用户故事**:
```

  - 优先级: P0 | 工作量: XL

  - 优先级: P0 | 工作量: L
```

### 8.7 F-10: 线程回复

**概述**: 用户可以对频道中的任意消息发起线程化回复，将特定话题的讨论从主频道中分离出来，避免主频道被密集讨论刷屏。线程是 Slack 核心用户体验的关键组成部分。

**交互流程**:
1. 在频道消息列表中，任意消息悬停时显示 "回复" 按钮（对话气泡图标）
2. 点击 "回复" → 右侧线程面板打开
3. 线程面板顶部显示父消息内容（作为上下文）
4. 面板中间显示该线程的所有回复消息列表（按正序排列）
5. 面板底部有输入框，用于发送线程回复
6. 发送线程消息 → 消息出现在线程面板中
7. 如果该频道中有 Agent → Agent 接收到线程消息上下文并自动响应
8. 主频道中该消息右下角显示 "N 条回复" 指示器
9. 点击面板外区域或关闭按钮 → 关闭线程面板

**业务规则**:
| 规则 | 内容 |
|------|------|
| 线程深度 | 线程内不支持嵌套线程（线程内消息不可再开线程），保持平面 |
| 上下文范围 | Agent 在线程中只接收线程内的消息作为上下文，而非整个频道 |
| 回复指示器 | 主频道消息在有新线程回复时更新 "N 条回复" 计数 |
| 消息类型 | 线程消息与主频道消息共享同一个 messages 表，通过 `thread_parent_id` 关联 |
| 通知行为 | MVP 阶段不实现独立通知，线程新回复不主动通知（需用户自行关注） |
| 持久化 | 线程消息同样持久化到数据库，支持历史查看 |
| 跨设备 | 线程状态同步（在一台设备上关闭线程，不影响其他设备） |

**线程与频道消息的关系**:
- 每个线程归属于一个父消息（通过 `thread_parent_id` 字段）
- 线程消息与频道消息是同一实体，区别仅在于是否有 `thread_parent_id`
- 查询频道消息时默认排除线程消息（只获取 `thread_parent_id IS NULL` 的消息）
- 查询线程时按 `thread_parent_id` 筛选并按 `created_at` 正序排列
- 当有新线程消息时，主频道通过 WS 事件 `thread.reply` 通知更新计数器

**线程面板 UI**:
| 状态 | 表现 |
|------|------|
| 加载态 | 线程面板骨架屏，父消息占位 + 消息列表灰色条 |
| 空态 | 面板显示父消息 + "还没有回复，开始讨论" 占位提示 |
| 回复列表 | 按时间正序显示，每条消息格式与频道消息一致 |
| 有新回复 | 面板中实时出现新消息（通过 WS 推送） |
| 错误态 | 面板内提示加载失败 + 重试按钮 |
| 关闭面板 | 点击 X 或点击面板外区域，面板收起，主频道恢复全宽 |

**用户故事**:
```

  - 优先级: P0 | 工作量: M
```

### 8.8 F-11: @提及 Agent

**概述**: 用户在发送消息时可以使用 @ 符号提及特定的 Agent 或用户。被提及的 Agent 会被单独触发并响应，即使该频道中有多个 Agent 或 Agent 默认不自动响应所有消息。这是类 Slack 对话体验的核心交互模式。

**交互流程**:
1. 用户在频道或 DM 输入框中输入 `@` 字符
2. 输入框上方弹出自动完成下拉列表，列出频道中所有可提及的成员（Agent + 用户）
3. 继续输入过滤字符 → 列表动态过滤匹配的成员
4. 使用鼠标点击或键盘（上下箭头 + Enter）选择目标
5. 选中后，输入框中显示 `@成员名称`（高亮显示）
6. 发送消息 → 消息正文中保留 `@成员名称` 文本
7. 后端解析消息中的 @提及 → 定位被提及的目标
8. 如果被提及的是 Agent →
   - Agent Daemon 收到触发信号（标记为 @提及 触发）
   - Agent 生成回复（可基于 "被 @提及" 调整行为）
   - 回复出现在频道中
9. 如果被提及的是用户 → MVP 阶段仅文本展示，不发送通知
10. 其他未被提及的 Agent（如果有）→ 不被触发（除非同时触发模式）

**提及解析规则**:
| 规则 | 内容 |
|------|------|
| 触发语法 | `@` + 成员名称（Agent 名称或用户名） |
| 匹配规则 | 全词匹配，区分大小写或不区分（MVP 采用大小写不敏感匹配） |
| 多提及 | 一条消息可提及多个 Agent，全部触发 |
| 自身提及 | Agent 提及自己 → 忽略，防止循环 |
| 解析时机 | 消息发送时由后端解析，不依赖前端格式 |
| 持久化 | 消息正文存储为纯文本 `@名称`，不存储特殊 mention ID |
| 跨频道 | 只能提及当前频道/DM 中的成员 |

**自动完成下拉 UI**:
| 状态 | 表现 |
|------|------|
| 加载态 | 输入 @ 后即时显示，无加载延迟（列表数据已预加载） |
| 有结果 | 显示匹配的成员列表，每行包含头像（或图标）+ 名称 + 类型标签（Agent/User） |
| 无结果 | 显示 "没有匹配的成员" 提示 |
| 继续输入 | 列表实时过滤，仅显示名称包含输入字符的成员 |
| 选中 | 点击或 Enter → 插入 `@名称`，下拉消失 |
| 取消 | 按 Escape 或点击输入框外区域 → 下拉消失 |
| 键盘导航 | 上下箭头移动高亮，Enter 确认选中 |

**通知行为**:
- MVP 阶段：@提及 仅用于触发 Agent，不涉及推送/邮件通知
- @提及 触发的 Agent 回复时，消息体中会注明 "（被 @提及 触发）" 以区别于自动响应
- 如果频道中 Agent 已开启自动响应，且消息同时包含 @提及 该 Agent → 按 @提及 逻辑处理（不重复触发）

**用户故事**:
```
US-15: 作为用户，我希望在消息中 @提及 Agent，以便手动触发指定 Agent 进行响应
  - 优先级: P0 | 工作量: M

US-16: 作为用户，我希望输入 @ 时看到自动完成列表，以便快速选择和提及成员
  - 优先级: P0 | 工作量: S
```

**验收标准**:
```gherkin

  Given 频道 "general" 中有 Agent "CodeReviewer"
  When 我发送消息 "@CodeReviewer 请审查这段代码"
  Then 消息成功发送
  And 后端解析出 @提及 CodeReviewer
  And Agent "CodeReviewer" 被触发
  And CodeReviewer 生成回复
  And 回复出现在频道 "general" 中

Scenario: @提及 自动完成下拉
  Given 我在频道 "general" 的输入框中
  When 我输入 "@"
  Then 弹出成员选择列表
  And 列表包含频道中所有 Agent 和用户
  And 列表按类型分组显示

Scenario: @提及 过滤
  Given 自动完成下拉已弹出
  When 我继续输入 "Code"
  Then 下拉列表过滤为仅名称包含 "Code" 的成员
  And 非匹配成员被隐藏

Scenario: @提及 仅触发被提及的 Agent
  Given 频道中有两个 Agent "Helper" 和 "Reviewer"

  And "Helper" 不被触发

  And 不产生循环触发

Scenario: @提及 不存在的成员

  Then 消息成功发送
  And 后端解析 @NonexistentAgent → 未匹配到任何成员
  And 不触发任何 Agent
  And 消息正常显示
```

### 8.9 F-20: 私信 (Direct Message)

**概述**: 私信（DM）是用户与其他用户或 Agent 之间的 1对1 私密对话通道。DM 在概念上是频道的一种特殊类型，但其行为和 UI 有显著区别。DM 是 Slack 类协作平台的核心组成部分。

**DM 与频道的区别**:
| 维度 | 频道 | DM |
|------|------|----|
| 参与者 | 多成员（2+ 人 + Agent） | 仅两人（用户-用户 或 用户-Agent） |
| 可见性 | 所有频道成员可见 | 仅对话双方可见 |
| 创建方式 | 手动创建 | 自动创建（选择收信人即创建） |
| 列表位置 | 侧栏上方频道区域 | 侧栏下方 DM 区域 |
| 成员管理 | 可添加/移除成员 | 不可添加/移除成员（固定两人） |
| 命名 | 有名称（如 #general） | 无名称，显示为对方名称 |
| Agent 行为 | 自动监听频道消息 | Agent 仅在被 @提及 或用户发送消息时响应 |
| 删除 | 创建者可删除 | 任意一方可删除（MVP: 暂不实现删除） |

**交互流程**:
1. 左侧栏底部显示 "私信（DM）" 区域标题
2. DM 区域列出已有的 DM 对话（显示对方名称、头像、最后消息预览）
3. 点击 DM 区域右上角的 "+" 按钮 → 弹出 "新私信" 选择对话框
4. 在对话框中搜索用户或 Agent（支持名称搜索）
5. 选中目标 → 创建或进入已有 DM 频道
6. DM 视图与频道视图一致：消息列表 + 输入框
7. 发送消息 → 对方（用户或 Agent）实时接收
8. 如果对方是 Agent → Agent 自动响应（行为同频道中的 Agent 响应）
9. 支持线程回复和 @提及（行为与频道一致）
10. DM 列表按最近消息时间降序排列，有新消息的 DM 排在顶部

**DM 列表 UI**:
| 状态 | 表现 |
|------|------|
| 加载态 | DM 区域显示灰色条状骨架屏 |
| 空态（无私信） | 显示 "还没有私信" + "发起新私信" 引导按钮 |
| 有 DM 列表 | 每个 DM 条目显示对方头像、名称、最后消息预览（截断 50 字）|
| 新消息 | DM 条目加粗 + 蓝色圆点指示未读 |
| 选中态 | DM 条目高亮，对应 DM 视图显示在右侧 |
| 错误态 | 加载失败 + 重试按钮 |

**DM 创建流程**:
1. 点击 "+" 按钮
2. 弹出 Modal/Dialog，包含搜索输入框
3. 输入名称 → 列出匹配的用户和 Agent
4. 列表显示名称、类型标签（用户/Agent）、在线状态
5. 点击选中 → 如果已有 DM 则直接进入，否则创建新 DM
6. 创建时，在 channels 表中插入一条 `type = 'dm'` 的记录
7. 自动将双方添加为频道成员
8. 跳转到该 DM 视图

**业务规则**:
| 规则 | 内容 |
|------|------|
| DM 去重 | 同一对用户之间只存在一个 DM 频道（双向唯一） |
| DM 标识 | channels 表增加 `type` 字段：'channel' | 'dm' |
| 可见性 | 用户只看到自己参与的 DM |
| Agent DM | 用户与 Agent 的 DM — Agent 自动响应所有消息 |
| 用户 DM | 用户与用户的 DM — 仅消息传递，无 Agent 参与（除非频道中有 Agent） |
| 最后消息 | DM 列表显示最后一条消息的预览（截断 50 字符）|
| 排序 | DM 列表按最后活动时间降序 |
| 批量加载 | DM 列表首次加载 20 条，滚动加载更多已归档的 DM |

**用户故事**:
```

  - 优先级: P0 | 工作量: M

US-18: 作为用户，我希望在侧栏看到 DM 列表，以便快速进入已有私信对话
  - 优先级: P0 | 工作量: S
```

### 9.1 性能指标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 消息端到端延迟 | P95 < 500ms | 从客户端发送到其他客户端收到的时间差 |
| API 响应时间（非实时）| P95 < 200ms | 服务端请求处理时间 |
| Agent 首 token 延迟 | P95 < 3s | Agent 触发到第一个 token 推送的时间 |
| WebSocket 连接数 | 单节点支持 1000 并发连接 | 压测 |
| 消息历史查询 | P95 < 500ms | 50 条消息的游标分页查询 |
| 页面加载时间 | FCP < 2s, LCP < 3s | Lighthouse 测试 |
| 系统可用性 | 99.5% | 服务端 uptime 监控 |

### 9.2 安全需求

| 类别 | 要求 |
|------|------|
| 认证 | JWT access token (15min expiry) + refresh token，bcrypt 密码存储 |
| 授权 | 用户只能访问自己的频道和消息；API 鉴权中间件逐请求检查 |
| 输入校验 | 所有用户输入 sanitize，SQL 注入防护（参数化查询），XSS 防护（输出编码） |
| API 安全 | Rate limiting，请求体大小限制（消息最大 100KB） |
| WebSocket 安全 | 连接时 JWT 校验，消息来源校验（不能伪造 sender_id） |
| 密码安全 | bcrypt hash，最少 8 位 |
| 安全性标头 | CSP, X-Frame-Options, X-Content-Type-Options |

### 9.3 可用性需求

| 类别 | 要求 |
|------|------|
| 前端响应 | 所有操作在 1s 内给出反馈；乐观更新优先 |
| 错误提示 | 所有错误有用户可读的中文提示 |
| 空态引导 | 每个空列表有明确的下一步引导 |
| 加载态 | 列表加载使用骨架屏，避免白屏闪烁 |
| 断线处理 | WebSocket 断线后显示"重新连接中..."，自动重连 |
| 消息失败 | 发送失败保留在输入框或显示重试按钮，不丢失内容 |
| 响应式 | 最小支持 1024px 宽屏，不要求移动端适配 |

### 9.4 可扩展性需求

| 类别 | 要求 |
|------|------|
| 数据库 | PostgreSQL，预留后续分表的可能性 |
| 服务端 | 无状态水平扩展（WebSocket 通过 Hub 模式管理） |
| Daemon | 独立进程，可以独立扩缩容 |
| 前端 | 按路由懒加载，WebSocket 与 REST 分离 |
| 配置 | 所有环境配置通过环境变量注入 |

### 9.5 可观测性

| 类别 | 要求 |
|------|------|
| 日志 | 结构化 JSON 日志（slog），包含 request_id |
| 健康检查 | /healthz (liveness) + /readyz (readiness) |
| 指标 | 请求数、延迟、错误率、WebSocket 连接数（Prometheus） |
| 错误追踪 | 关键错误输出到日志，配合 slog level 管理 |
